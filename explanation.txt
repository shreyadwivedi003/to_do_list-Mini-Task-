To-do List Api implementation:

1.) In memory array
For this project, I chose to store data using a standard, in-memory JavaScript array (`const notes = []`). Since this is a mini-task focused on mastering routing and HTTP methods, using an array let me skip the setup overhead of database connection drivers (like MongoDB). The obvious drawback is that this data is volatile; whenever the Express server restarts, the array clears to empty. For a production app, I would use such data that is persistent.

2.) Using unique Ids
Initially, the endpoints targeted notes using their position in the array (`/notes/:index`). However, I realized this creates a massive data corruption flaw. If you delete an item at index 0, every single item behind it immediately shifts down one spot. If a client is interacting with a UI that hasn't refreshed instantly, sending an update or delete request to a specific index could target the wrong note entirely.
To fix this, I introduced a global `nextId` counter. Now, every new note gets stamped with a permanent, unique numeric ID that increments sequentially. Because this ID is static, a note's ID never changes when neighboring items are shuffled or deleted.

3.)Array integrity
In early drafts, I considered using the JavaScript `delete` keyword (e.g., `delete notes[index]`). But doing this leaves an "empty hole" (`undefined`) at that position instead of shrinking the array container itself, which messes up count tallying. 
By moving to unique IDs, my new implementation uses `.findIndex()` to locate the targeted note's position, and then applies `.splice(noteIndex, 1)`. Splice cleanly slices the record completely out of the array and shifts subsequent items forward, maintaining a clean, contiguous collection.

4.) Preventing data corruption on partial updates
A critical flaw with naive update handlers is that if a user submits a PATCH request to change *only* the title, the missing description field might get assigned as `undefined`, inadvertently wiping out existing description data. 
To achieve a true partial update, my PATCH handler explicitly checks whether `title` or `description` are undefined before overwriting them (`if (title !== undefined)`). If the client leaves a field out of the request payload, the existing value remains safely untouched.

APIs that have been used here are:
1. GET api: To get/read all the notes that have been pushed to backend from frontend and keep their count as well.
2. POST api: To create a new note.
3. PUT api: To partially or completely update a note that already exists
4. DELETE api: Used to delete an existing note
